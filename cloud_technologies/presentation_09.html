Лекція №9 (2 години)

Хмарні технології

План лекції

  • Концепція спостережуваності (Observability): метрики, логи, трейси
  • Хмарні сервіси моніторингу: CloudWatch, Azure Monitor, Cloud Monitoring
  • Логування та централізований збір журналів
  • Розподілене трасування
  • Відмовостійкість: патерни надійних систем

Важливість спостережуваності

Розгортання системи в хмарі — це лише початок. Дійсна зрілість хмарних операцій визначається здатністю команди зрозуміти, що відбувається всередині системи в будь-який момент часу.

Паралельно, будь-яка система врешті-решт зазнає збоїв — питання не «чи відбудеться відмова», а «коли» і «наскільки добре система відновиться».

Observability vs Monitoring

Різниця в підходах

Натисніть "вниз", щоб побачити детальне порівняння

Monitoring (Моніторинг)

Monitoring — відповідає на питання «Чи справна система?». Ми заздалегідь знаємо, що перевіряти.
  • Перевіряємо відомі метрики та показники
  • Аларми на попередньо визначені пороги
  • Чорна скринька — ми знаємо, що дивитися

Observability (Спостережуваність)

Observability — відповідає на питання «Чому система поводиться саме так?». Дозволяє досліджувати невідомі раніше стани.
  • Система є спостережуваною, якщо її внутрішній стан можна визначити лише на основі зовнішніх виходів
  • Дозволяє аналізувати неочікувані проблеми
  • Біла скринька — ми можемо дослідити будь-який стан

Три стовпи спостережуваності

Metrics, Logs, Traces

Натисніть "вниз", щоб побачити детальний опис кожного стовпа

Metrics (Метрики)

Числові значення, що вимірюються у часі:

  • CPU utilization: 78%
  • Requests/sec: 1 250
  • Error rate: 0.3%
  • Response time (p95): 230 ms
  • Queue depth: 42

Метрики — агреговані, займають мало місця, ідеальні для алертів і дашбордів

Logs (Журнали)

Дискретні події у часі:

{
  "timestamp": "2024-01-15T14:23:45.123Z",
  "level": "ERROR",
  "service": "payment-service",
  "message": "Payment gateway timeout",
  "transaction_id": "tx-9f8b2c",
  "user_id": "u-12345",
  "duration_ms": 5001,
  "gateway": "stripe"
}

Логи детальні, але займають багато місця. Ідеальні для дебагінгу та розслідування інцидентів

Traces (Розподілені трейси)

Фіксація шляху запиту через усі мікросервіси:

HTTP GET /api/orders/123
│
├── auth-service (25ms) — перевірка токена
│
├── order-service (180ms) — обробка
│   ├── db-query (45ms) — SELECT * FROM orders
│   └── inventory-service (120ms) — перевірка наявності
│       └── db-query (35ms)
│
└── notification-service (15ms) — надсилання email

Трейси критичні для мікросервісних архітектур

SLA, SLO, SLI

Ієрархія показників якості

Натисніть "вниз", щоб побачити детальні визначення

SLI (Service Level Indicator)

SLI — конкретна метрика, що вимірює якість обслуговування
  • Доступність: успішні запити / всі запити × 100%
  • Затримка: % запитів, що виконані швидше порогового часу
  • Частота помилок: помилкові запити / всі запити

SLO (Service Level Objective)

SLO — цільове значення SLI
  • Доступність ≥ 99.9% за місяць
  • p99 затримка ≤ 500 мс
  • Частота помилок ≤ 0.1%

SLA (Service Level Agreement)

SLA — юридично обов'язкова угода між провайдером і клієнтом, базується на SLO. Передбачає компенсацію при порушенні.

Error Budget (Бюджет помилок)

SLO: 99.9% доступність / місяць
Error budget: 100% - 99.9% = 0.1% = 43.8 хвилин простою/місяць

Використано: 25 хвилин → залишок: 18.8 хвилин

Команда може витрачати error budget на ризиковані деплойменти

Amazon CloudWatch

Центральна платформа моніторингу та спостережуваності в AWS

  • Metrics: автоматичний збір метрик від усіх AWS-сервісів
  • Alarms: сповіщення при перевищенні порогів
  • Dashboards: візуалізація метрик у реальному часі
  • Logs: централізований збір журналів
  • Container Insights: моніторинг EKS/ECS

  Threshold: > 80% протягом 3 з 3 хвилин
  Action:    → SNS notification (email/SMS)

Переваги: легке парсинг, фільтрування, агрегування та запити

Рівні логування

  • DEBUG: детальна діагностична інформація (тільки при розробці)
  • INFO: нормальні події (запуск, успішна дія)
  • WARN: потенційно небезпечна ситуація, але система продовжує працювати
  • ERROR: помилка, що перешкоджає виконанню конкретної операції
  • CRITICAL/FATAL: критична помилка, що може спричинити зупинку сервісу

Практики керування логами

Retention Policy (Термін зберігання):

  • DEBUG-логи: 7 днів
  • INFO-логи: 30 днів
  • ERROR-логи: 90 днів
  • Аудит-логи (безпека): 1–7 років (GDPR, SOX)

Log Sanitization (Приховування PII):

# Неправильно — логування картки
logger.info(f"Processing card {card_number}")  # '4532-1234-5678-9012'

# Правильно — маскування
logger.info(f"Processing card ****{card_number[-4:]}")  # '****9012'

Розподілене трасування

У мікросервісній архітектурі один HTTP-запит може пройти через 10–20 сервісів. Розподілене трасування відстежує весь шлях запиту та вимірює час виконання на кожному кроці.

Terminology:

  • Trace — весь шлях одного запиту від початку до кінця
  • Span — одна операція в межах trace
  • Trace ID — унікальний ID, що передається між сервісами

OpenTelemetry — відкритий стандарт

OpenTelemetry (OTel) — відкритий стандарт та набір SDK для збору метрик, логів і трейсів, що підтримується CNCF.

Переваги OTel:

  • Vendor-neutral: один код → будь-який бекенд
  • Підтримується всіма хмарними провайдерами
  • Стає стандартом де-факто для observability в K8s

Відмовостійкість

Design for Failure

Принцип «Everything can fail» стверджує: будь-який компонент системи врешті-решт відмовить. Задача архітектора — спроєктувати систему так, щоб відмова одного компонента не призводила до відмови всієї системи.

Chaos Engineering — навмисне введення збоїв для перевірки стійкості

Circuit Breaker (Автоматичний вимикач)

Патерн, що запобігає каскадним відмовам:

Стан: CLOSED (нормальний)
  → сервіс B недоступний: помилки
  → поріг помилок перевищено

Стан: OPEN (відкритий)
  → запити до B відхиляються миттєво (fallback)
  → чекаємо час відновлення (30 сек)

Стан: HALF-OPEN
  → пробний запит до B
  → успіх → CLOSED
  → невдача → OPEN знову

Retry з Exponential Backoff та Jitter

import random
import time

def call_with_retry(func, max_retries=3):
    for attempt in range(max_retries):
        try:
            return func()
        except TransientError as e:
            if attempt == max_retries - 1:
                raise
            # Exponential backoff + random jitter
            wait = (2 ** attempt) + random.uniform(0, 1)  # 1s, 2s, 4s + jitter
            time.sleep(wait)

Jitter запобігає "thundering herd" проблемі

Інші патерни відмовостійкості

  • Bulkhead — ізоляція ресурсів між незалежними частинами системи
  • Timeout — завжди встановлюйте таймаут для зовнішніх викликів
  • Cache-Aside — читання з кешу перед зверненням до БД
  • Graceful Degradation — спрощений варіант відповіді при відмові сервісу

Multi-AZ та Multi-Region

Multi-AZ (всередині регіону):

  • Захист від відмови одного датацентру
  • Автоматичне переключення: RDS Multi-AZ, ALB між AZ
  • Синхронна реплікація даних

Multi-Region (між регіонами):

  • Захист від катастрофи цілого регіону
  • Складніша реплікація (асинхронна або активно-активна)
  • Набагато вища вартість та складність

Runbook та Incident Response

Runbook — задокументована покрокова інструкція для реагування на конкретний тип інциденту. Критично важлива для ефективного відновлення в 3 годині ночі. 
Incident: High Error Rate on Payment Service (> 5%)
Severity: P1

1. Перевірити CloudWatch Dashboard: payment-service-prod
2. Перевірити останні деплойменти (CodeDeploy Console, останні 2 год)
3. Перевірити залежності: Stripe API Status
4. Перевірити логи: CloudWatch Insights → payment-service → рівень ERROR
5. Якщо деплоймент — виконати rollback
6. Якщо Stripe → увімкнути резервний провайдер через feature flag
7. Повідомити stakeholders через Slack #incidents
8. Провести Post-Mortem протягом 48 годин

Висновки

  • Observability — ширша концепція, ніж моніторинг. Три стовпи (метрики, логи, трейси) разом дають повну картину стану системи
  • SLI/SLO/SLA та Error Budget — фундамент SRE-практик. Помилковий бюджет балансує між надійністю та швидкістю розробки
  • Хмарні платформи моніторингу надають готові інтеграції з усіма сервісами. Для мультихмарних — Prometheus+Grafana або Datadog
  • Розподілене трасування є незамінним для мікросервісних систем, де єдиний запит охоплює десятки сервісів
  • Патерни відмовостійкості та Multi-AZ розгортання — фундамент production-систем. Design for Failure та Chaos Engineering гарантують стійкість
