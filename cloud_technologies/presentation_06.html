Лекція №6

План лекції

  • Три типи хмарного сховища
  • Блочне сховище (Amazon EBS)

  • Файлове сховище та NAS у хмарі (EFS, FSx)

1. Три парадигми зберігання

Порівняння архітектур

Блочне (Block) Об'єктне (Object) Файлове (File)
  • Як жорсткий диск.
  • Зберігає дані блоками.
  • Підключається до одного сервера.
  • (Для ОС, Баз Даних).
  • Як "бездонна флешка" в Інтернеті.
  • Файл + Метадані. Доступ по HTTP.
  • Безмежний обсяг.
  • (Для фото, відео, бекапів).
  • Як мережевий диск (NAS).
  • Ієрархія папок і файлів.
  • Одночасний доступ для тисяч серверів.
  • (Спільні файли, веб-сайти).

2. Блочне сховище (EBS)

Ваш віртуальний жорсткий диск

Amazon EBS (Elastic Block Store)

  • Живе окремо від сервера. Сервер вимкнули — диск лишився.
  • Автоматично створює резервну копію всередині свого датацентру.
  • Snapshots (Знімки): Інкрементальні бекапи диску (зберігають лише зміни), які автоматично відправляються в S3.
  • Тип gp3 — золотий стандарт (дешевий і швидкий NVMe SSD).

3. Об'єктне сховище (Amazon S3)

Найпопулярніший сервіс AWS

Концепція S3

Дані існують не в папках, а в Кошиках (Buckets) як плоский набір об'єктів. Кожен об'єкт має свій унікальний URL.

  • Надійність: 99.999999999% (11 дев'яток). Дані зберігаються одразу в трьох датацентрах.
  • Може віддавати статичні сайти (HTML/CSS) прямо з S3 без жодного сервера!

Класи зберігання (Як зекономити?)

Клас Застосування
Standard Частий миттєвий доступ (аватарки користувачів). Найдорожчий.
Standard-IA Рідкий доступ, але коли треба — видає миттєво (звіти за минулий місяць). Удвічі дешевше.
Glacier Deep Archive "Цифрова мерзлото". Архівування на роки. Очікування файлу — 12-48 годин. Але ціна — копійки ($0.00099/ГБ).

* Lifecycle Policies вміють переміщувати файли між цими класами автоматично.

4. Файлове сховище у хмарі

Amazon EFS та FSx

  • Amazon EFS (Elastic File System): NFS-диск для Linux. Можна змонтувати одночасно до сотень серверів. Розмір "гумовий" — росте автоматом.
  • Amazon FSx for Windows: Рідний мережевий диск для серверів на Windows (підтримує протокол SMB і Active Directory).
  • FSx for Lustre: Супершвидка система для машинного навчання (ML) та суперкомп'ютерів.

5. Аварійне відновлення (DR)

Disaster Recovery

Дві головні метрики

RPO (Recovery Point Objective): "Скільки даних ми готові втратити?" (Наприклад: максимум за 15 хвилин до аварії).

RTO (Recovery Time Objective): "Як довго система може 'лежати'?" (Наприклад: відновити сайт за 1 годину).

Стратегії відновлення

  1. Backup & Restore: Бекапи лежать в архіві. Якщо сервер впаде — створюємо новий, качаємо бекап. Дешево, але дуже повільно (RTO = години).
  2. Pilot Light: База даних уже працює в резервному місті, сервери вимкнені і чекають.
  3. Warm Standby: Зменшена копія проекту працює 24/7. При аварії просто збільшуємо її потужність.
  4. Multi-Site (Active/Active): Сайт одночасно працює в Європі та США. Якщо Європа падає, трафік непомітно переходить до США. Супердорого, але RTO = 0.

Висновки

  • EBS (блочне): для баз даних та ОС (один сервер).
  • S3 (об'єктне): для файлів, медіа та бекапів. Нескінченне і дешеве.
  • EFS (файлове): для спільного доступу сотень серверів до однієї папки.
  • Життя без бекапів — біль. Snapshots та S3 Glacier роблять резервне копіювання дешевим та надійним.
