#### 1.2 Термінал та CLI (Command Line Interface)

Робота розробника не обмежується написанням коду в редакторі. Левова частка інструментів (системи контролю версій, пакетні менеджери, лінтери, локальні сервери) керується не через візуальний інтерфейс кнопок, а через введення спеціальних текстових команд.

##### 1.2.1 Вступ до командного рядка: PowerShell (Windows) / Bash (Linux/macOS)

**CLI (Command Line Interface)** — це текстовий інтерфейс користувача для управління операційною системою та програмами. На противагу GUI (Graphical User Interface — інтерфейсу з вікнами та кнопками), CLI споживає мінімум ресурсів комп'ютера і дозволяє автоматизувати складні рутинні завдання шляхом написання скриптів.

У різних операційних системах використовуються різні командні оболонки (Shell):

- **Bash (Linux / macOS):** Історичний стандарт індустрії. Більшість серверів у світі працюють під управлінням Linux, тому знання базових команд Bash є критично важливою навичкою (так званий _Unix-way_).
- **PowerShell (Windows):** Сучасна, потужна об'єктно-орієнтована оболонка від Microsoft. Хоча PowerShell має власний набір команд (cmdlets), він підтримує багато "аліасів" (псевдонімів), які роблять його схожим на Bash для базових операцій.

> 🖼️ [МІСЦЕ ДЛЯ СКРИНШОТУ: Відкритий інтегрований термінал у нижній панелі VS Code, де з правого боку у випадному списку обрано оболонку "PowerShell" або "bash", а в робочому полі очікує рядок вводу команди]

##### 1.2.2 Навігація файловою системою (команди `cd`, `ls`/`dir`, `mkdir`, `rm`)

Щоб успішно працювати в терміналі, ви маєте чітко розуміти свою поточну робочу директорію (Working Directory). Команди виконуються в контексті тієї папки, де ви зараз "знаходитесь".

**Базові команди навігації та управління (Unix/Windows):**

1.  **Перегляд вмісту папки (List Directory):**
    - `ls` (в Unix) або `dir` (в Windows/PowerShell — хоча `ls` там теж працює як аліас). Показує всі файли та папки в поточній директорії.
2.  **Зміна поточної директорії (Change Directory):**
    - `cd НазваПапки` — перейти всередину папки.
    - `cd ..` — піднятися на один рівень вгору (у батьківську папку).
    - `cd ~` — перейти у вашу домашню директорію користувача (наприклад, `C:\Users\Student\`).
3.  **Створення нової папки (Make Directory):**
    - `mkdir назва_проєкту` — створить нову папку у поточній директорії.
4.  **Створення порожнього файлу:**
    - `touch index.html` (Unix) або `New-Item index.html` (PowerShell).
5.  **Видалення файлів та папок (Remove):**
    - `rm файл.txt` — видалити файл.
    - `rm -r папка/` (Unix) або `Remove-Item папка -Recurse` (PowerShell) — рекурсивно видалити папку з усім її вмістом. **Увага:** файли видаляються назавжди, минаючи "Кошик".

> 🖼️ [МІСЦЕ ДЛЯ СКРИНШОТУ: Термінал із послідовністю введених команд розробника (наприклад: `mkdir new-site`, `cd new-site`, `touch index.html`, `ls`) та відображеним результатом їх виконання (список з одним файлом `index.html`)]

##### 1.2.3 Запуск адміністративних скриптів та обробка помилок терміналу

Часто під час спроби виконати команду (наприклад, запуск локального сервера `npm start`) студенти зіштовхуються з помилками. Як їх обробляти:

- **Command Not Found (Команду не знайдено):** Означає, що операційна система не знає такої програми. Зазвичай це виникає, якщо програма не встановлена, або шлях до неї не додано у змінну середовища системного шляху (`PATH`). Переконайтеся, що під час встановлення ПЗ ви відмітили чекбокс "Add to PATH".
- **Execution Policies (Політика виконання в Windows):** За замовчуванням PowerShell може блокувати запуск локальних скриптів (наприклад, при спробі активувати віртуальне середовище Python `venv`). Для вирішення проблеми необхідно запустити термінал від імені Адміністратора та виконати команду `Set-ExecutionPolicy RemoteSigned`.
- **Як зупинити завислий скрипт:** Якщо ваш локальний сервер запущено, або програма нескінченно виконується (зависла в циклі), поставте фокус у вікно терміналу і натисніть комбінацію клавіш **`Ctrl + C`**. Це надішле системний сигнал скасування (SIGINT) і поверне контроль над рядком вводу.

##### 1.2.4 Використання Git Bash

Оскільки веб-розробка тісно пов'язана з Unix-подібними серверами, користувачам Windows часто бракує рідних Linux-утиліт (наприклад, `grep`, `awk`, `curl`, `ssh`).

**Git Bash** — це застосунок для середовища Windows, який емулює термінал Linux (оболонку Bash) та надає набір стандартних інструментів GNU. Він встановлюється автоматично разом із системою контролю версій Git.
Рекомендується використовувати саме Git Bash як термінал за замовчуванням у VS Code на операційній системі Windows, щоб мати уніфікований досвід розробки (однакові команди незалежно від того, на Mac, Linux чи Windows ви працюєте).

> 🖼️ [МІСЦЕ ДЛЯ СКРИНШОТУ: Вікно терміналу Git Bash у VS Code на Windows, де введено команду `ls -la` і консоль демонструє типовий кольоровий Linux-вивід списку файлів з їхніми правами доступу]

##### 1.2.5 Використання Docker та Docker Compose

У сучасній розробці замість того, щоб встановлювати бази даних, інтерпретатори та веб-сервери напряму у свою операційну систему (що призводить до "забруднення" системи та конфліктів версій), використовують контейнеризацію.

**Docker** — це платформа, яка дозволяє запакувати ваш додаток та всі його залежності у стандартизований блок — **контейнер**. На відміну від важких віртуальних машин (VirtualBox чи VMWare), контейнери Docker ділять ядро вашої ОС, тому вони запускаються за долі секунди та споживають мінімум оперативної пам'яті.

**Базові команди CLI для управління контейнерами:**

- `docker ps` — переглянути список контейнерів, що зараз запущені та працюють.
- `docker logs <ім'я_контейнера>` — подивитись логи (консольний вивід) застосунку всередині ізольованого контейнера.
- `docker-compose up -d` — підняти одразу всю інфраструктуру проєкту (наприклад, бекенд на Node.js + базу даних MySQL + кеш Redis) у фоновому режимі (detached), описану у файлі `docker-compose.yml`.
- `docker-compose down` — зупинити та видалити всі контейнери проєкту.

> 🖼️ [МІСЦЕ ДЛЯ СКРИНШОТУ/СХЕМИ: Схема концептуальної різниці між традиційною Віртуальною Машиною (Hardware -> Host OS -> Hypervisor -> Guest OS -> App) та Docker-контейнером (Hardware -> Host OS -> Docker Engine -> App)]

##### 1.2.6 WSL (Windows Subsystem for Linux)

Для студентів, які працюють на Windows, але потребують абсолютно нативного Linux-середовища (наприклад, для складних Docker-контейнерів або роботи з бібілотеками, які погано компілюються на Windows), розроблено **WSL 2**.

Це підсистема Windows для Linux, розроблена спільно Microsoft та Canonical. Вона дозволяє запускати повноцінне ядро Linux (зазвичай дистрибутив Ubuntu) безпосередньо у Windows без використання повільних віртуальних машин або подвійного завантаження (Dual Boot).

**Переваги для розробника:**

1. Файлова система Linux (`EXT4`) працює значно швидше з величезною кількістю дрібних файлів (що критично для папок типу `node_modules`).
2. Всі серверні інструменти та скрипти працюють рівно так само, як вони будуть працювати на справжньому "бойовому" сервері під час деплою.
3. VS Code має офіційне розширення **"WSL"**, яке дозволяє тримати графічний інтерфейс редактора на Windows, але "під капотом" виконувати термінал та запуск коду безпосередньо в ізольованому середовищі Linux.

> 🖼️ [МІСЦЕ ДЛЯ СКРИНШОТУ: Екран розробки, де у VS Code відкрито термінал WSL (наприклад, Ubuntu). Зліва знизу на синьому фоні Status Bar видно статус підключення: `>< WSL: Ubuntu`]
