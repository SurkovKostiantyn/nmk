#### 3.2 Технічне документування проєкту

Код розповідає комп'ютеру, що робити, а документація розповідає людям, навіщо ви це зробили і як цим користуватися. У сучасному IT існує принцип **"Documentation as Code"** — підхід, при якому документація створюється тими ж інструментами, що й код (Markdown, Git, автоматичні генератори).

##### 3.2.1 Використання Markdown для створення поглибленої документації

Хоча `README.md` є обов'язковим, великі проєкти потребують розширеного опису архітектури, гайдів для розробників та інструкцій користувача.

Інструменти для структурованої документації:

- **GitHub Wiki** — вбудований у репозиторій інструмент для створення бази знань. Дозволяє створювати ієрархію сторінок без "засмічення" основної гілки коду;
- **Папка docs/** — стандартний спосіб зберігання документації всередині репозиторію. Це дозволяє версіонувати документацію разом із кодом;
- **Статичні генератори (Docusaurus, Docsify)** — інструменти, які перетворюють ваші Markdown-файли на повноцінний сучасний веб-сайт із пошуком та навігацією.

##### 3.2.2 Основи автогенерації документації (Swagger для API)

Написання документації для API (список маршрутів, параметрів, відповідей) вручну — це шлях до помилок, оскільки код змінюється швидше за текст. **Swagger (OpenAPI)** вирішує цю проблему шляхом автоматичної генерації опису інтерфейсу безпосередньо з вашого коду.

Переваги Swagger:

- **Інтерактивність** — клієнт може протестувати будь-який запит до сервера прямо в браузері через зручний інтерфейс;
- **Стандартизація** — формат OpenAPI розуміють майже всі сучасні інструменти розробки;
- **Завжди актуально** — при зміні назви поля в коді, воно автоматично зміниться і в документації.

##### 3.2.3 Ведення CHANGELOG.md: фіксація ключових змін

**CHANGELOG.md** — це файл, у якому в хронологічному порядку ведеться список усіх значущих змін у кожній версії проєкту. Це критично важливо для користувачів та інших розробників.

Стандарт **"Keep a Changelog"** рекомендує групувати зміни за такими категоріями:

- **Added** — для нових функцій;
- **Changed** — для змін у вже існуючому функціоналі;
- **Deprecated** — для функцій, які скоро будуть видалені;
- **Removed** — для функцій, які вже видалені;
- **Fixed** — для виправлених багів;
- **Security** — у випадку виправлення вразливостей.

Приклад запису:

```markdown
## [1.0.1] - 2026-05-14

### Fixed

- Виправлено помилку з валідацією email у формі реєстрації.

### Added

- Додано підтримку темної теми для головного екрана.
```

Такий підхід дозволяє швидко зрозуміти, що нового з'явилося в проєкті, не переглядаючи сотні комітів у Git.
