# Практична робота №6 (2 години)

**Тема**: Налагодження коду (Debugging).
Використання React DevTools та точок зупинки (breakpoints) у браузері для пошуку помилок у стані та пропсах.

**Мета**: Навчитися виявляти та усувати логічні помилки у React-додатках без використання застарілого методу `console.log`; опанувати роботу з розширенням React Developer Tools для інспектування ієрархії компонентів, стану (`state`) та пропсів (`props`); навчитися ставити точки зупинки (Breakpoints) у вкладці Sources браузера Chrome для покрокового виконання JavaScript-коду.

**Необхідні інструменти**: Браузер (Google Chrome або Edge), встановлене офіційне розширення [React Developer Tools], локально запущений проєкт з попередніх лабораторних/практичних робіт (`npm run dev`), редактор VS Code.

## План заняття

1. Встановлення та знайомство з інтерфейсом React Developer Tools (вкладки Components та Profiler).
2. Штучне створення логічної помилки у компоненті (з попередніх робіт).
3. Використання панелі Components для інспектування розбіжностей у State та Props.
4. Знайомство з панеллю Chrome DevTools (Sources) для налагодження коду.
5. Встановлення Breakpoints та покрокове виконання (Step-by-step) функції-обробника подій.

## Хід виконання роботи

### 1. Налаштування React DevTools

Після встановлення розширення React Developer Tools у браузер і перезапуску сторінки вашого проєкту, відкрийте панель інструментів розробника (F12).
Знайдіть дві нові вкладки: **⚛️ Components** та **⚛️ Profiler**.

Перейдіть у вкладку **Components**. Вона показує не HTML-вузли (як вкладка Elements), а саме дерево React-компонентів:
`App` -> `Dashboard` -> `StatisticsSection` -> `StatBadge`.

### 2. Створення штучного багу (помилки) в існуючому коді

Візьмемо ваш компонент "Стрічка новин" (NewsFeed) або "Авторизація" (Login) з попередніх робіт в єдиному репозиторії.
Зробимо навмисну логічну помилку в `AuthContext` або компоненті профілю для подальшого пошуку.

Відкрийте компонент, який відображає список і має кнопку (наприклад, видалення або лайк поста).
Змініть робочий код обробника на такий (з помилкою неправильного звернення до властивості):

```jsx
// Штучна помилка: ми хочемо передати id, але випадково передаємо весь об'єкт події (event)
const handleLikeClick = (e) => {
  // e - це об'єкт SyntheticBaseEvent, а не ID поста!
  setLikedPosts((prev) => [...prev, e]);
};

// ... у рендері:
<button onClick={handleLikeClick}>Лайк</button>;
```

### 3. Пошук помилки (State/Props) через вкладку Components

1. Запустіть додаток (`npm run dev`) відкрийте сторінку в браузері.
2. Натисніть кнопку "Лайк". Візуально нічого не змінилося або додаток зламався (краш).
3. Якщо ви використовували `console.log`, вам би довелося йти в код і додавати його. ЗАМІСТЬ ЦЬОГО:
   - Відкрийте панель **⚛️ Components**.
   - Знайдіть ваш компонент у дереві (використовуйте поле пошуку ліворуч).
   - Натисніть на нього. У правій колонці (Sidebar) ви побачите поточний `props`, `hooks` (зокрема State).
4. Розгорніть `State`. Ви побачите, що замість очікуваного масиву `[1, 2, 5]` (де цифри — це ID постів), у вас лежить масив з гігантським об'єктом `Синтетичної події`.
   **Висновок:** DevTools миттєво показав розбіжність між очікуваним станом пам'яті та реальним.

### 4. Пошук помилки за допомогою Breakpoints (Sources)

Тепер знайдемо місце в коді, _де саме_ цей невірний об'єкт туди потрапляє.

1. Перейдіть до стандартної вкладки браузера **Sources**.
2. Відкрийте файл вашого компонента. (У Vite-додатку файли зазвичай лежать у `localhost:5173/src/...`). Також можна скористатися гарячою клавішею `Ctrl + P` (або `Cmd + P`) і написати назву вашого компонента, наприклад `NewsFeed.jsx`.
3. Знайдіть рядок коду `setLikedPosts(prev => [...prev, e]);`.
4. Клацніть зліва від номера цього рядка, щоб поставити **блакитну позначку (Breakpoint)**.
5. Натисніть кнопку "Лайк" на сайті ще раз.
6. Браузер **ЗУПИНИТЬ ВПЛИВ ЧАСУ** (Paused in debugger).
7. Наведіть курсором миші на змінну `e` прямо в коді на вкладці Sources. Браузер покаже tooltip з усіма даними всередині змінної у цей конкретний момент часу.
8. Ви зрозумієте, що `e` — це подія.

### 5. Виправлення коду та перевірка

Щоб виправити код, обробник має отримувати саме `postId`:

```jsx
const handleLikeClick = (postId) => {
  setLikedPosts((prev) => [...prev, postId]);
};

// У JSX стрілочна функція "перехоплює" подію e, і викликає функцію з правильним аргументом
<button onClick={() => handleLikeClick(post.id)}>Лайк</button>;
```

Після виправлення коду у VS Code, перейдіть в React DevTools (вкладка Components) і переконайтеся, що State тепер оновлюється правильними масивами `[id1, id2]`.

_(Примітка: для швидкої зупинки виконання коду без вкладки Sources, розробник може написати слово `debugger;` прямо в коді React-компонента і зберегти файл)._

## Завдання для самостійного виконання

1. У вашому єдиному репозиторії лабораторних робіт оберіть найбільш складний компонент, який працює зі станом або формами (наприклад, `ProfilePage` або форма реєстрації).
2. За допомогою React DevTools (вкладка Components) знайдіть цей компонент, і вручну (натиснувши на значення у правій панелі розширення) змініть значення його State або Props "на льоту" (без перевантаження сторінки). Зафіксуйте, як миттєво відреагує UI.
3. Додайте у функцію отримання даних (`useEffect` з викликом API) команду `debugger;`.
4. Перезавантажте сторінку. Коли браузер зупиниться, скористайтеся інструментами "Step Over" (F10) та "Step Into" (F11) у панелі Sources, щоб покроково пройтись по рядках `fetch` (або `axios`) запиту.
5. Дослідіть панель "Scope" (Область видимості) під час зупинки, щоб побачити всі локальні та глобальні змінні, доступні компоненту в цей момент.

## Контрольні запитання

1. В чому полягає критична перевага використання React Developer Tools порівняно зі звичайним інспектуванням DOM-елементів (вкладка Elements) при пошуку помилок у React-додатку?
2. Поясніть алгоритм дій розробника при встановленні та використанні Breakpoint у вкладці Sources браузера Chrome. Що відбувається з виконанням програми в цей момент?
3. Яке ключове слово (резервоване в JavaScript) можна написати безпосередньо у вихідному коді компонента, щоб змусити браузер автоматично зупинити виконання при відкритій панелі розробника?
4. У вкладці Components React DevTools у правій панелі є розділи "hooks" та "props". Навіщо розробнику може знадобитися можливість ручної зміни цих значень прямо в браузері "на льоту" під час дебагу?
5. Що означають дії "Step Over", "Step Into" та "Step Out" у панелі управління дебагером браузера (Sources) під час зупинки виконання коду?
