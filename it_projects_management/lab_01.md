# Лабораторна робота №1 (2 години)

**Тема:** Аналіз бізнес-кейсів та ініціація IT-проєкту

Опанування навичок старту проєкту: обґрунтування його доцільності (Business Case), правильна постановка цілей (SMART) та створення головного установчого документа — Project Charter (Статут проєкту).

**Мета:** Навчитися ідентифікувати бізнес-потреби замовника, формулювати мету проєкту за критеріями SMART та розробляти базовий документ ініціації (Статут проєкту) для забезпечення спільного розуміння цілей усіма стейкхолдерами.

**Технологічний стек:** Google Docs / Microsoft Word / Confluence (для документування), Miro / Draw.io (за бажанням, для візуалізації).

---

## Завдання

Вам потрібно обрати ідею для IT-проєкту (це може бути навчальний проєкт, стартап або автоматизація існуючого бізнесу). Наприклад: "Мобільний застосунок для замовлення кави заздалегідь", "Система обліку відвідуваності студентів", "Telegram-бот для психологічної підтримки". Для обраної ідеї необхідно виконати кроки ініціації.

### Перелік завдань:

1. Описати бізнес-кейс (проблему та варіанти вирішення).
2. Сформулювати цілі проєкту за критеріями SMART.
3. Розробити Project Charter (Статут проєкту).

---

## Хід виконання роботи

### Крок 1. Розробка Бізнес-кейсу (Business Case)

Бізнес-кейс — це документ, який пояснює **навіщо** компанії витрачати гроші на цей проєкт (який прибуток він принесе або яку проблему вирішить).

Опишіть вашу ідею за такою структурою:

1. **Проблема / Можливість (Problem Statement):** Що зараз працює погано? (Наприклад: "Студенти витрачають по 15 хвилин у черзі в буфет на перерві, через що запізнюються на пари і буфет втрачає прибуток").
2. **Пропоноване рішення:** Що конкретно ви пропонуєте розробити алгоритмічно чи програмно?
3. **Очікувана вигода (Benefits):** Фінансова (збільшення прибутку на Х%) або нефінансова (покращення іміджу, економія часу).
4. **Альтернативи:** Що буде, якщо НІЧОГО не робити? Або чи можна вирішити проблему без написання коду (наприклад, просто найняти ще одного касира)?

### Крок 2. Формування цілей за SMART

Мета "Зробити класний додаток" — це погана мета. Мета повинна відповідати п'яти критеріям SMART. Розпишіть головну мету вашого проєкту:

- **S (Specific - Конкретна):** Що саме ми робимо? (Створити мобільний застосунок на React Native для iOS та Android...)
- **M (Measurable - Вимірна):** Як ми зрозуміємо, що досягли успіху? (...який дозволить обробляти 500 замовлень на день...)
- **A (Achievable - Досяжна):** Чи реально це зробити вашою командою? (Так, у нас є 3 розробники).
- **R (Relevant - Значуща):** Чому це важливо для бізнесу сам зараз? (...щоб розвантажити живі черги в час пік...)
- **T (Time-bound - Обмежена в часі):** До якого дедлайну? (...до 1 вересня поточного року).

_Напишіть 1-2 цілі для вашого проєкту у форматі SMART._

### Крок 3. Створення Project Charter (Статут проєкту)

Статут проєкту — це документ, який офіційно "запускає" проєкт та наділяє PM-а повноваженнями.
Складіть Статут для вашого проєкту, заповнивши наступні пункти:

1. **Назва проєкту:**
2. **Спонсор проєкту:** (Хто платить гроші? Змоделюйте цю особу/компанію).
3. **Керівник проєкту (PM):** (Ваше ПІБ).
4. **Мета проєкту:** (Скопіюйте з кроку 2).
5. **Високорівневі вимоги (High-level Requirements):** Який базовий функціонал має бути в релізі (перерахуйте 3-5 головних фіч).
6. **Ключові стейкхолдери:** (Хто найбільше зацікавлений у проєкті: розробники, юзери, адміністрація).
7. **Високорівневі ризики:** Що може завадити старту або завершенню проєкту? (Вкажіть 2-3 ризики, наприклад: нестача фінансування, технічні обмеження застарілого API).
8. **Критерії успіху (Success Criteria):** Наприклад, "Додаток опубліковано в AppStore, отримано 1000 встановлень за перший місяць".

## Контрольні запитання

1. У чому принципова різниця між документами _Business Case_ та _Project Charter_?
2. Наведіть власний приклад цілі, яка НЕ відповідає критерію "Measurable" (Вимірна), і перепишіть її так, щоб вона стала вимірною.
3. Хто зазвичай затверджує (підписує) Project Charter: керівник проєкту (PM), команда розробників чи Спонсор проєкту? Чому?
4. Для чого в Статуті проєкту (на самому початку, коли ще немає коду і розкладу) вказують високорівневі ризики?
5. Назвіть щонайменше 3 обов'язкових розділи, які мають бути у Project Charter.
